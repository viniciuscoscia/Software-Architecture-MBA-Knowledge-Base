## Contexto
Bom, pessoal, neste capítulo eu quero falar com vocês sobre Tracing. E como tudo isso funciona e o porquê isso é praticamente obrigatório, se você for trabalhar com microsserviços. E o porquê que eu estou dizendo isso? Se você trabalha com Tracing isso significa que você vai conseguir rastrear todas as requisições que caem no seu microsserviço.

Vamos imaginar que o microsserviço 1 recebe uma requisição, e ele depende do microsserviço 2, que manda uma requisição para o 3, para o 4, para o 5. O que a gente aprende aqui no momento que isso acontece? Que cada requisição vai ter uma latência aqui, um response time de cada microsserviço. Por quê? Porque você vai ter que trafegar os dados pela rede. Existem alguns pontos que podem acontecer durante esse processo. Normalmente, dois pontos importantes que você tem que se ligar. O primeiro ponto é que o usuário final aqui faz uma requisição, (deixa eu botar o usuário aqui bonitinho para a gente). Esse usuário aqui faz uma requisição, e essa requisição demora para caramba. E você vai querer saber, por que está demorando? Será que é o microsserviço 1 que realmente está lento? Será que é o microsserviço 2 que está lento? Então, o microsserviço 2, o microsserviço 1 tem que esperar o 2? Será que é o microsserviço 3 que está lento? Então, o 2 tem que esperar o 3, e o 1 tem que esperar o 2? Será que é o 4 que está lento? E todos esses tem que esperar? Será que é o 5? Então, isso é um problemão. É um baita problema, porque você tem que conseguir ter esse tipo de noção. E o que eu mais vejo nos dias de hoje, é gente subindo a arquitetura baseada em microsserviços e ter esse tipo de problema, e não ter a menor ideia de onde isso está acontecendo.

E a gente tem um outro “problemaço” aqui também. Qual é esse problema? Eu chamei esse microsserviço, e o usuário final recebe um erro 500 na ponta. Recebeu um erro 500 na ponta, a grande pergunta que fica aqui é, quem teve o erro? O microsserviço 1 está com problema, é o 2 que está com problema, é o 3, o 4 ou o 5? Começa por esse ponto. Agora, teve o problema? Qual é o problema ele teve? Quais são os dados que ele recebeu para que esse problema acontecesse? Então, não sei se você percebe, mas quando nós estamos trabalhando aqui com microsserviços, você tem uma baita dificuldade de saber e localizar problemas que venham a ocorrer durante as requisições. A gente não tem essa dificuldade quando a gente está trabalhando com sistemas monolíticos, porque se o usuário recebe um problema, o problema está dentro do seu sistema. Provavelmente, se você for olhar no log, o log vai te mostrar ali um stack trace ou alguma coisa assim do que deu errado. O grande ponto é, se você recebe uma requisição que deu problema, e o problema está aqui no microsserviço 5, você vai ter que olhar o log do microsserviço 1, do 2, do 3, do 4, e para perceber que o erro aconteceu apenas no 5. E, às vezes, o erro que aconteceu no 5 não foi necessariamente o erro dessa request que o usuário acabou trabalhando. 

Então você consegue perceber que o nível aqui é praticamente exponencial da dificuldade que você vai ter de conseguir descobrir e fazer o troubleshooting da sua aplicação. É por conta disso que a gente vai falar de tracing e mostrar para você como o tracing pode te ajudar. Então, vamos falar sobre isso no nosso próximo vídeo com mais calma aqui. Vamos nessa!

## Correlation ID
Bom, pessoal, no vídeo anterior eu falei do problema que a gente pode ter caso nós não tenhamos acesso muito simples, de uma forma muito clara para a gente conseguir identificar problemas nos nossos microsserviços. Uma das formas de você conseguir minimizar e pelo menos ter um pouco mais de facilidade para identificar esses problemas é trabalhando com tracing. Só para você saber, tracing é um dos três pilares da observabilidade. 

Observabilidade é uma prática que você tem três pilares, logs, métricas e tracing. Onde o log você pega sobre o que aconteceu, mostra o que aconteceu no seu sistema. As métricas mostram naquele momento o que está acontecendo no seu sistema, quantos acessos, quantas requisições, quanto você tem de CPU, quantas compras e coisas desse tipo. E o tracing é basicamente a resposta do que acontece de ponta a ponta de uma requisição. Então, quando esse usuário manda uma requisição que começa a ser estendida para diversos microsserviços, eu preciso ter uma maneira de rastrear essa requisição. E para que a gente consiga fazer, nós damos um ID para cada requisição. Então, quando eu sei que esse ID bate aqui nessa requisição, eu falo o seguinte: “olha, dessa requisição para cá o ID é 1”. Assim, o que vai acontecer? Dessa requisição para cá o ID é 1 também, essa aqui o ID é 1, e assim vai. Isso significa que se eu souber o ID de cada requisição, eu sei que quando der problema, eu sei que a requisição do usuário X bateu no 1, no 2, no 3, porque eu sei que essas requisições têm o mesmo ID, então, elas vão se complementar. Então, isso é importante.

Normalmente, esse ID que a gente fala… chamamos isso de Correlation ID, que é o ID que a gente passa entre as requisições para que nós possamos fazer essa correlação das nossas requisições. Então, todas as vezes que você ouvir falar em Correlation ID, você vai perceber que a gente está querendo utilizar uma forma de identificar uma requisição. E que ela vai ser transversal ali, de forma geral, entre todos ou uma parte dos seus microsserviços, e se você tiver um problema, você vai ter mais facilidade para rastrear. Então, o nosso primeiro passo aqui é conseguir identificar uma request de ponta a ponta. O nosso primeiro passo aqui vai ser a gente conseguir implementar o Correlation ID. E isso tem que ser de forma intencional, ou seja, toda vez que você fizer uma requisição e você receber uma primeira requisição, se você não tiver ali com você um Correlation ID, você gera um. Assim, quando você mandar essa requisição para um outro microsserviço, você vai mandar essa requisição e vai mandar esse Correlation ID para essa requisição também. E esse segundo microsserviço, o que vai fazer? Ele vai verificar se ele está recebendo o Correlation ID, e se ele estiver recebendo, a requisição que ele for fazer, vai mandar junto com ele esse mesmo Correlation ID. Então, esse Correlation ID vai sempre ter que ser extraído de cada requisição e reenviado a cada requisição para garantir que você não vai ter esse tipo de problema. Então, isso sempre vai acontecer de forma intencional. Se você não fizer essa extração e não enviar esse Correlation ID, você não vai conseguir fazer, completar esse tipo de coisa.

Hoje em dia, já existem diversas bibliotecas, diversas formas e técnicas para você fazer isso. Normalmente, você cria middleware, para você não ter que ficar repetindo o código o tempo inteiro em relação a isso. Mas de qualquer forma, saiba que isso é importante. E falando em bibliotecas, no próximo vídeo, eu quero falar com você sobre OTEL, que é Open Telemetry. E o projeto da Open Telemetry nos ajuda a fazer exatamente isso aqui. Então, vamos nessa!

## Tracing e Collector
Bom, pessoal, vamos falar do Open Telemetry aqui. Só uma coisa para você saber, toda vez que você vê aqui Otel  é OpenTelemetry. Isso aqui é um ponto importante. Então, eu não sei se você percebeu, mas, aqui em um dos apresentações anteriores, eu falei que a gente tinha que ter o Correlation ID para os dados serem passados aqui para cada microsserviço. Quando você está usando OpenTelemetry, você ainda vai continuar fazendo isso. Você, obrigatoriamente, tem que fazer isso, ou seja, passar esse Correlation ID. O grande ponto é que a OpenTelemetry provê um SDK que facilmente você consegue fazer isso. E esse SDK tem para todas quanto é linguagem. Tem para .NET, tem para Java, tem para Go, tem para Javascript etc. Depois, basta você entrar lá no site da OpenTelemetry, ali na documentação, e ver como você aplica o SDK para a linguagem que você trabalha no dia a dia.

Agora, uma outra coisa que o OpenTelemetry faz aqui para a gente, é o seguinte, ele tem um conceito de algo que a gente chama de Collector. O que isso significa? Significa que esse Collector é um coletor de dados de telemetria, ou seja, ele é um coletor de logs, de métricas e também de tracing. Isso significa que para cada requisição que eu for fazendo aqui, eu, aqui nesse SDK, configuro para que os dados que eu estou trabalhando fiquem sendo enviados para esse coletor. Então, todos os meus microsserviços vão se comunicar e mandar essas informações nesse coletor. Ou seja, esse coletor centraliza essas informações. Ou seja, ele é um servidor que acaba trabalhando como um outro microsserviço. Também existem algumas variações, onde você coloca esse coletor do lado de cada microsserviço, ou seja, como um sidecar, por exemplo, e, assim, todo o microsserviço fica mandando os dados para o coletor.

Agora, qual é a vantagem de você trabalhar com esse coletor? É que você consegue fazer uma série de configurações e você fala o seguinte, “olha, para tracing eu quero que os dados sejam enviados para o Jaeger e para o Zipkin”. “Ah, eu quero que para os logs os dados sejam enviados para o New Relic”. “Ah, eu quero que para a métrica os dados sejam enviados ali para o Prometheus”. Então, fazendo isso, os seus microsserviços mandam para o coletor e na configuração do seu coletor, você escolhe para qual vendor as informações vão ser enviadas. E se, por algum momento, você quiser trocar de vendor, você mexe apenas no arquivo de configuração aqui desse coletor. Para que você possa escolher para qual vendor… qual novo vendor que você está alterando e que você vai querer mandar essa informação. Então, isso aqui é extremamente rico, extremamente poderoso e, hoje em dia, a maioria dessas aplicações nas grandes empresas e etc. já estão nascendo dessa forma. Todas enviando os dados para um coletor e esse coletor mandando os dados para o back-end ou para o vendor que foi configurado. Aqui, somente para você ter uma ideia, é como fica um processo de tracing aqui. Os dados caíram no microsserviço demo, esse microsserviço demo chamou o microsserviço 2, o microsserviço 2 chamou o microsserviço 3. Então, eu consigo ver de cabo a rabo a minha requisição. Então, essa requisição aqui 1 demorou quase 1 segundo, essa mais 2, essa mais 2, então 5 segundos no total aqui para essa requisição. E aqui você consegue ver, inclusive, a árvore de chamada de um microsserviço para o outro, ou seja, o microsserviço 1 chama o 2 que chama o 3. Tudo isso acontece e para a gente ter essas informações de tracing aqui, a gente tem essas informações porque eu estou utilizando o Correlation ID e essas informações aqui foram enviadas através de um colector para a gente conseguir mandar os dados aqui, nesse caso, para o Jaeger, um software para fazer análise de tracing.
