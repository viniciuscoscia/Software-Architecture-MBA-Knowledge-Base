# Introdução a EDA
Bom, pessoal, agora a gente vai falar de algo muito, muito, muito importante, que é uma forma de você conseguir trabalhar, é uma arquitetura que visa você a trabalhar baseado em eventos, então isso aqui é chamado de Event Driven Architecture ou o famoso EDA. 

E como funciona o Event Driven Architecture? Ela é uma arquitetura baseada em eventos, então, a primeira coisa que a gente tem que entender aqui é o que são eventos e como esses eventos funcionam. A primeira coisa que você tem que se atentar é que um evento é algo que aconteceu no seu sistema. Um evento é sempre algo no passado, não existe um evento no futuro, existe sempre um evento no passado. Então, aqui eu tenho alguns exemplos de eventos: compra aprovada, a porta do carro abriu, o site foi acessado, tudo isso são dados de eventos. Por quê? Porque são dados que já aconteceram e em nossos sistemas, normalmente, muitos eventos acontecem a todo momento, cliente cadastrado, dados alterados, e-mail enviado… a gente tem diversas informações sobre eventos.

Agora, um ponto importante é que você tem que se atentar que existem alguns tipos de eventos em nossos sistemas, e esses eventos vão mudar a estratégia que você vai ter de comunicação entre os sistemas que vão produzir os eventos e os sistemas que vão ler esses eventos. Então, é importante a gente entender os tipos desses eventos. E o que eu vou trazer aqui para vocês no nosso próximo vídeo é quais são os tipos de eventos e quando normalmente esses tipos de eventos são utilizados, porque muitas pessoas acabam ficando na dúvida de como produzir os eventos, quando vale a pena produzir qualquer um desses três tipos, vamos dizer, de eventos que a gente estava trabalhando. Vamos ver isso no próximo vídeo para ficar mais claro para todo mundo. Vamos nessa!
# Event Notification
Bom, pessoal, a gente vai falar agora sobre tipos de eventos e você vai perceber que esses tipos de eventos mudam completamente o jogo da forma como você vai trabalhar. Você vai perceber que você usa esses tipos de eventos normalmente em casos de uso diferentes. Então, vamos falar sobre esses tipos de eventos e quando esses caras vão nos ajudar no dia a dia. Então, nós temos o nosso primeiro tipo de evento, esse tipo de evento se chama Event Notification.

O que é um Event Notification? Um Event Notification, normalmente ele vai trazer dados mínimos sobre algum evento, obviamente, que aconteceu. Então, vamos imaginar o seguinte, eu tenho aqui, Order ID 1, status aprovado, ou seja, isso é uma notificação. Vamos imaginar que eu tenho uma compra que aconteceu num sistema e a compra foi aprovada. Existem muitos sistemas que querem saber quando uma compra foi aprovada. Vamos imaginar o sistema da nota fiscal, o sistema do frete, o sistema do centro de distribuição etc. Então, nesse caso aqui o Event Notification é importante porque a gente vai mostrar quando um estado da nossa informação muda. O Event Notification normalmente trabalha muito com estado, quando algo muda no sistema ele notifica sistemas que têm interesse nesse tipo de informação.

O porquê trabalhar com Event Notification é importante ? Porque a gente tem a vantagem de ter dados muito pequenos e quando nós temos sistemas muitos microsserviços, no ecossistema muito grande, a gente tem, por exemplo, Mercado Livre, (a gente sempre dá exemplos aqui do Mercado Livre). Os caras ali, têm ali milhões e milhões de mensagens sendo processadas por minuto, é quase um bit de mensagem processadas por minuto sobre eventos que aconteceram no sistema deles. Agora imagina se tudo que você fosse mandar de evento fosse uma quantidade de dados muito grande. Vamos imaginar se esse evento tivesse muita informação. Ia fazer com que o custo de processamento dos sistemas fossem muito grandes, a rede ia ficar muito mais congestionada. Então, nesse tipo de caso, quando o estado de algo muda, você consegue enviar essas informações e essas informações sendo enviadas os sistemas podem tomar as suas decisões. Isso é o Event Notification e você vai perceber que é muito utilizado, principalmente quando você quer saber mudanças de estado. No próximo vídeo, a gente vai para o segundo tipo de evento.
# Event carried state transfer
Bom, pessoal, agora a gente vai falar sobre um outro tipo de evento, chamado de Event Carried State Transfer. Para que serve esse cara, como esse cara funciona. Bom, esse tipo de evento, no final das contas, é um evento de stream. O que isso significa? Significa que ele é um evento que carrega todos os dados que aconteceram. Então, diferentemente do Event Notification Only, onde o que importa para gente é a mudança de estado, no Event Carried State Transfer, o que importa para a gente não é apenas a mudança de estado, mas sim o dado em si.

Vamos imaginar que a gente tem uma compra que aconteceu, e a compra, assim que foi aprovada, a gente tem que emitir a nota fiscal. No caso de uma nota fiscal, apenas falar que o status foi aprovado não vai resolver. Por quê? Porque para emitir a nota fiscal, eu preciso saber o nome do produto, os dados de compra, sei lá, o CPF, o imposto que eu vou pagar, o preço que foi colocado… eu preciso saber um monte de informação. Então, esse tipo de informação não é só uma mudança simples de estado, esse tipo de informação traz o dado em si, porque com esse dado ele pode executar o processamento e a execução de uma tarefa. Esse tipo de evento, muitas vezes, é necessário, porque sem os dados completos, os sistemas não conseguem fazer o processamento da forma como ele trabalha. Mas normalmente você vai ver que esse tipo de cara só é utilizado quando essas informações são realmente necessárias, porque quando elas não são, você vai utilizar  Event Notification.

É muito comum, quando você for trabalhar com sistemas, você trabalhar de forma híbrida. Você vai trabalhar tanto com Event Notification quanto com Event Carry State Transfer, porque em certos momentos você precisa dos dados completos, mas em outros momentos você precisa apenas da mudança de estado. E, isso vai trazer, de forma muito mais inteligente, as decisões que você vai tomar quando os sistemas produzirem eventos e quando os sistemas precisam dessas informações. Então sempre se lembre, quando você utilizar um e outro, qual é o grande problema de muita gente, de muitos sistemas, que quando a volumetria muda? Eles acabam trazendo problemas. É que muitos sistemas acabam trabalhando apenas dessa forma, com Event Carry State Transfer. E o que acontece? É muito dado sendo trafegado na rede e, muitas vezes, esses dados não são tão necessários. Quando a gente começa a ter um volume muito grande nesses sistemas, simplesmente a rede congestiona, os sistemas ficam mais lentos, sendo que o Event Notification já  resolveria. É importante você ter essa consciência para que, de forma intencional, você escolha quando e qual tipo de evento vai ser lançado.

Eu acho que o mais interessante de tudo, principalmente desses conceitos que eu estou trazendo para você, é que a implementação de como você vai fazer isso é o de menos. Mas o importante é que você tome decisões de forma intencional e quando você toma decisões de forma intencional, você consegue ser muito mais assertivo. E para você tomar decisões intencionais, você tem que saber quais são as opções que você tem. E é aqui que eu estou trazendo as opções, para que você consiga trabalhar da forma mais otimizada possível nos seus sistemas. Então, vamos nessa, porque a gente tem mais uma forma de a gente trabalhar. Vamos nessa!
# Event sourcing
Bom, pessoal, agora a gente vai falar do nosso terceiro tipo de evento, chamado de Event Sourcing. Como o Event Sourcing funciona? A utilização dele é bem específica. A utilização dele nos ajuda a guardar o histórico de tudo que aconteceu no nosso sistema, ou seja, vamos imaginar que eu emito um evento falando que a compra foi aprovada, então, você vai perceber aqui que parece que eu tenho um Event Notification, e aqui, nesse caso, eu até tenho, porém, você vai perceber que esse Event Notification, além de ter essas informações, normalmente tem um Time Stamp, ou se ele não tiver, você vai perceber que os eventos são guardados numa determinada ordem. Porque cada vez que esse evento ocorre, esses dados são persistidos em um banco de dados, então, você vai perceber aqui, a compra foi aprovada e os dados foram guardados. O status. O produto foi enviado, é gerado mais um dado, mais uma linha no banco de dados. O produto foi entregue, mais uma linha no banco de dados. Ele consegue versionar para a gente cada evento, sendo possível a gente fazer o replay.

O que isso significa? Que se eu ler esse banco na ordem certinha que os dados foram colocados, dá na mesma para o meu sistema consumidor. Porque a gente tem cada mudança de estado que o sistema vai acontecendo. E isso para a gente, é muito importante em alguns casos de uso específico. Quais são esses casos de uso? Um exemplo muito claro é auditoria. Porque se eu consigo ver cada mudança de evento, eu consigo ver cada coisa que aconteceu dentro do meu sistema, ou resultante ali do meu sistema. E, nesse caso, a gente consegue tomar uma decisão. A gente consegue deixar muito claro o que aconteceu.

Eu vou dar um exemplo para vocês. Vamos imaginar que eu estou num extrato bancário, e eu tenho um saldo bancário ali no meu banco. Você acha que toda vez que o sistema vai calcular o saldo, ele pega todos os débitos e todos os créditos e gera o saldo todas as vezes? Você tem 20 anos de conta no banco e ele vai fazer essa conta, essa matemática, cada vez para você trabalhar dessa forma? Claro que não. Normalmente, eles vão trabalhar com alguns snapshots, que mostra o seu saldo, e depois de tempos em tempos ele vai fazendo essa conta ali para você. Agora, já pensou se cada vez que eu mudasse o meu saldo, eu não lançasse, eu não fizesse o lançamento do que aconteceu do meu débito, do meu depósito? Se alguém chega e muda ou dá um erro no sistema onde ele adiciona o saldo de um milhão, e o cara não tem um milhão, o cara tava negativo, como você vai saber o saldo final desse cara? Bom, se você tiver todos os eventos que resultam numa alteração de saldo, basta pegar o seu sistema, apagar o saldo final e mandar ele reprocessar todas essas mudanças de estado, que assim você vai chegar naquele saldo final de forma correta. Então, é muito interessante para a auditoria; É muito interessante para você conseguir pegar erros no sistema, porque você sabe o que está acontecendo; É muito importante para você conseguir dar o replay, e você garante muito mais segurança.

Mas vamos imaginar que o cara troca o endereço de entrega, ao invés de você apagar do banco de dados o endereço antigo e colocar o novo, você adiciona o endereço novo e os dados antigos ficam ainda registrados no banco de dados. Existe um caso muito interessante de um banco de dados, que é chamado de Datomic. E esse Datomic é um banco de dados que hoje, se eu não me engano, virou propriedade do Nubank. O Nubank comprou a empresa que fez o Datomic e etc. E o que o Datomic faz? Ele é um banco de dados imutável, significa que tudo que acontece de mudança, todos os updates que acontecem, geram um novo registro, então, significa que por padrão, você faz Event Sourcing e você consegue ter essa auditoria. É muito interessante a forma como esse banco de dados trabalha, por exemplo. Então, uma coisa também em relação a Event Sourcing, você vai ver que ele é bem associado a algo que a gente chama de CQRS. A gente vai falar isso daqui a pouco, mas é importante você entender que a gente tem esses tipos de eventos. Então, vamos nessa!

# Schemas evolutivos
Bom, pessoal, agora a gente vai falar de algo assim, não é importante, é muito importante, que é chamado de esquema evolutivo. Por que eu estou dizendo isso, pessoal? Todas as vezes que a gente vai trabalhar com esses eventos, ou que a gente vai fazer com que a gente tenha um publicador numa ponta e um consumidor nessa outra ponta, você vai ter algo extremamente simples, algo que pode virar um caos na sua aplicação. É o esquema da mensagem.

Vamos olhar aqui. A gente tem, por exemplo, essa mensagem aqui onde eu falo "order ID é 1”, “status aprovado, product ID é 1” e etc. O sistema que publica sabe que ele tem que mandar essa mensagem nesse formato, mas o sistema que lê também tem que saber que a mensagem está nesse formato. O que acontece caso, por exemplo, esse sistema estivesse sempre mandando a mensagem com o product underline ID e simplesmente de um dia para o outro ele fizesse assim, mandasse product ID sem o underline. O que iria acontecer? O sistema que fosse ler essa mensagem não ia achar o product underline ID, não iria reconhecer o product ID e o que iria acontecer? Ia dar muito problema. O grande ponto é que conforme o tempo passa os sistemas mudam, os dados mudam e o esquema de publicação de uma mensagem muda também. Agora vamos imaginar que eu tenho 10 microsserviços lendo um microsserviço que está mandando uma mensagem. Já pensou se esse microsserviço começa a mandar a mensagem no formato diferente e aqueles 10 microsserviços começam a mandar. Então a gente precisa ter um meio de reenforçar o padrão de envio e recebimento das mensagens, desses eventos. E a forma que a gente faz isso, normalmente, (deixa eu voltar isso aqui) é através de esquemas, ou seja, você chega para o consumidor e para o publicador e define o esquema de envio de mensagens e o recebimento é esse aqui, e todo mundo vai seguir. O grande ponto é que quando esse esquema precisa mudar a gente precisa ter técnicas e mecanismos que a gente mantenha compatibilidade desses esquemas. E existem algumas formas de eu manter essa compatibilidade. Então vamos falar sobre os três tipos básicos para você conseguir evoluir o esquema, quando o seu sistema precisa ter essas mudanças de padrão.

A gente tem o primeiro formato que é chamado de Forward Compatibility. O que isso faz? Os dados são produzidos com um novo esquema, mas ainda mantém a compatibilidade de leitura com o antigo esquema, ou seja, não há alteração de código pelo consumidor. Vamos imaginar que eu tenho o sistema 1, ele manda informação, e o sistema 2 consegue ler. Depois o sistema 1 muda o padrão de mandar informação, mas o padrão que ele manda informação não quebra o sistema que está lendo, mas ele começa a trazer, a enviar novos dados. Então, isso é bem interessante. Isso é chamado de Forward Compatibility.

Mas a gente tem uma outra, um outro formato de você conseguir trazer essas informações, que é chamado de Backward Compatibility. O que significa? Que o dado que é produzido no esquema antigo pode ser lido como se fosse um novo esquema. Como assim? Vamos imaginar que eu tenho um consumidor, e esse consumidor está criando uma nova feature. Vamos imaginar que agora todas as vezes que um cartão de crédito for aprovado, a gente vai colocar 100 reais na conta da pessoa, ou um valor ‘X’ na conta. Então, eu avisei quem vai consumir com antecedência falando, “olha, gente, vai entrar uma nova feature daqui 15 dias”. E a gente vai começar a mandar um campo para vocês chamado bônus. Então, toda vez que tiver esse campo bônus, você já adiciona como crédito para a pessoa. Mas o que acontece? O consumidor conseguiu resolver esse problema com 10 dias de antecedência. Ou seja, se o consumidor quiser colocar isso em produção, ele consegue colocar em produção mesmo que o produtor ainda não esteja enviando essa mensagem. Isso vai ajudar bastante no trabalho ali no dia a dia.  Olha só que interessante que a gente consegue trabalhar. Ou seja, o consumidor consegue colocar uma nova feature mesmo antes dessa feature dos dados serem enviados para ele. Uma coisa interessante também é que, quando você trabalha com backward compatibility, mesmo você fosse trabalhar com event sourcing e as mensagens estivessem no padrão antigo, ainda assim você iria conseguir ler. Então, isso é importante.

Agora, a gente tem um outro formato, que é o terceiro formato, chamado de Full Compatibility. É a combinação dos dois mundos. Ele faz com que os esquemas antigos consigam ser processados, que você consiga prever as novas mudanças do esquema ao mesmo tempo. É muito difícil você conseguir fazer isso. Você tem que ter muito planejamento para você conseguir trabalhar, é possível, mas nem sempre isso é necessário. Porque você vai gastar um bom tempo de planejamento para conseguir trabalhar dessa forma. Mas de qualquer jeito, existem essas três formas de você conseguir trabalhar. Então, esse é o nosso esquema evolutivo.
# Schema registry
Agora a gente vai falar sobre schema registry, e tem tudo a ver com a parte de compatibilidade de schema. Essa imagem aqui é a imagem do funcionamento da Confluent schema registry. 

A Confluent schema registry é um schema registry que, utilizado em conjunto com Apache Kafka e, você vai entender o porquê isso é importante e faz bastante diferença para quem vai trabalhar. 

Aqui você tem o Kafka, ou seja, você publica mensagem no Kafka e ler mensagens no Kafka, ou seja, eu pego esses dados mando para o Kafka e recebo esses dados aqui, que são enviados para o Kafka. O grande ponto aqui para a gente é o seguinte, você consegue ter um serviço chamado schema registry. Você cria um registry, ou seja, você cria um padrão de mensagens. Esse padrão normalmente pode ser feito no padrão Avro, que é a espécie de um JSON mas que, antes de trazer os dados, fala quais são os tipos dos dados. Você consegue trabalhar com Protobuf, que ele vai mostrar para você qual é o schema do envio das mensagens ou no formato JSON.

Isso significa que eu crio uma versão do schema, e quando eu enviar uma mensagem o meu produtor, que vai enviar mensagem, vai ler o esquema, vai pegar o ID desse schema e mandar a mensagem. O Apache Kafka, para aceitar essa mensagem, vai saber que o formato dos dados tem que estar exatamente no formato do id daquele esquema. Ou seja, tem que estar baseado naquele schema. Se não estiver baseado naquele schema ele não vai deixar essa mensagem ser enviada. 

O grande ponto também é que para que o seu produtor não tenha que ficar, para cada mensagem, lendo esse esquema, ele vai gravar localmente no cache, para que ele não precise ficar enviando a todo momento. 

Do outro lado, o consumidor faz a mesma coisa. ele lê o esquema, vai receber os dados ali e agora ele sabe qual é o padrão dos dados que ele vai ter que serializar para ele conseguir processar a informação.

Vamos imaginar que esse schema evoluiu. Então, simplesmente vai ser gerado uma versão 2 aqui.  Quando o produtor for mandar uma mensagem o ID que ele vai mandar é o ID 2 , e o Kafka já vai saber qual é o formato. E o consumidor também vai saber qual é o formato. Assim,  trabalhar com o schema registry é interessante porque ele vai poupar um bom tempo, e vai poupar “brigas” entre times. Todo mundo colocou os esquemas lá no registry, e todo mundo vai seguir.  Quando surgir um novo esquema, gera bota lá no Registry e todo mundo vai seguir também. Então, isso é bem interessante e faz muito sentido essa utilização.

Nem todo mundo consegue trabalhar com schema registry dessa forma. Então, o que eu recomendo sempre, quando você não tem isso, é você fazer o seguinte: cria um repositório e nomeie. Crie os esquemas para você conseguir trabalhar. E você pode falar o seguinte “isso aqui é o nosso contrato, todo mundo que for trabalhar baixa esse repositório e utiliza esse contrato aqui”. Assim, pelo menos os dados não ficam soltos.  Eles ficam organizados dentro de um padrão e todo mundo vai ficar mais feliz e contente. Obviamente, com o schema registry você tem essas diversas vantagens que eu acabei de falar.

No schema, quando você criar um esquema, você vai escolher o tipo de compatibilidade também. Isso é bem interessante na hora que você vai trabalhar, e basicamente é isso. Então, tome muito cuidado porque tudo que gera desacoplamento entre sistemas, ou tudo que a gente fala como dados entre sistemas, quando a gente trabalha com sistemas distribuídos, a gente vai ter sempre esse entrave aí no meio, que são os padrões que vão evoluindo ao longo do tempo. Então, os schema registry vão te ajudar.
